# Linux学習記録 Day30-2
# ディスク監視システムの状態管理化とSlack通知の最適化
- cron監視の通知ノイズを減らし、運用可能な状態管理型監視へ改善するため
## 実践したこと
- ディスク使用量監視スクリプト（disk_monitor.sh）を状態管理型に改善
- 単純な閾値判定から「OK / WARN / CRITICAL」の3段階状態管理へ拡張
- 状態を/tmpのファイルに保存し、前回状態との差分で通知制御を実装
- Slack通知を状態変化時のみ送信するように変更し、通知ノイズを削減
- ログ出力を追加し、監視結果と状態遷移を追跡可能にした
## 実装内容
### 監視対象と閾値
- 監視対象：ルートディスク（/）
- 閾値：
  - 90%以上 → CRITICAL
  - 80%以上 → WARN
  - それ未満 → OK
### 状態管理の仕組み
前回状態をファイルに保存し、差分で通知を制御
000
STATE_FILE="/tmp/disk_state.txt"

if [ -f "$STATE_FILE" ]; then
    prev_state=$(cat "$STATE_FILE")
else
    prev_state="UNKNOWN"
fi
000
### 現在状態の判定
000
if [ "$USAGE" -ge 90 ]; then
    state="CRITICAL"
elif [ "$USAGE" -ge 80 ]; then
    state="WARN"
else
    state="OK"
fi
000
### 状態変化時のみ通知
000
if [ "$state" != "$prev_state" ]; then
    # Slack通知処理
fi
000
### Slack通知処理
状態ごとにメッセージを変更して送信
- CRITICAL → 緊急通知
- WARN → 警告通知
- OK → 復旧通知
### ログ管理
監視結果と状態をログに記録
000
echo "$(date) usage=${USAGE}% state=${state}" >> "$LOG_FILE"
000
## 改善前との違い
### 改善前
- 閾値を超えたら毎回通知
- 状態の概念なし
- ノイズが多い
### 改善後
- 状態ベース管理
- 変化時のみ通知
- 運用を意識した設計
## 発生した問題と対応
### 状態管理の導入
- 単純なファイル存在管理から状態文字列管理へ変更
- OK / WARN / CRITICALで状態を明確化
### 通知ノイズ問題
- 毎回通知される問題を解消
- 状態変化時のみSlack通知するよう改善
### ログ設計
- 監視結果と状態を記録することでデバッグ性向上
## 使用したコマンド
### 実行・確認

bash disk_monitor.sh
000
### 状態確認
000
cat /tmp/disk_state.txt
000
### ログ確認

000
## 学んだこと
- 監視スクリプトは「値」ではなく「状態」で管理する方が運用に向いている
- 状態をファイルに保存することで、シンプルにステート管理が可能になる
- 通知は「発生時」ではなく「状態変化時」に制御することでノイズを減らせる
- bashでも状態機械（ステートマシン）を実装できる
- 小規模でも監視・通知・ログの三点セットで運用設計になる
## まとめ
ディスク監視スクリプトを単純な閾値判定から状態管理型に進化させた。
これにより通知ノイズを削減し、運用を意識した監視システムとして改善できた。
bashだけでも状態管理・監視・通知を備えた小規模SRE的システムを構築できることを理解した。

- 状態管理によるイベント駆動型監視を実装
