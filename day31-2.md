# # Linux学習記録 Day30-2
# Docker + cron + ops-lab 再構築と運用設計の改善

本日は ops-lab 環境の再構築と、Docker・cron周りの不整合修正を行った。

本環境は単なる復旧ではなく、
「壊れることを前提に、再構築できる運用設計へ改善すること」を目的としている。

従来構成では以下の問題が発生していた
- cron設定がコンテナ再生成後に消失する
- entrypointとcronの責務が混在している
- UID固定によるビルドエラー（環境依存）
- ログ出力先がホスト・コンテナで不一致
- 通知が状態変化に関係なく発火しノイズ化する

これらを起点に、Docker / entrypoint / cron の責務を再設計した。

## 設計原則（今回の改善軸）

今回の再設計では以下を統一した

- 再現性（Reproducibility）
- 責務分離（Separation of Concerns）
- 状態管理（State-based Monitoring）
- 非永続性前提（Immutable Container）
- 運用ノイズ削減（Noise Reduction）
## 実施内容
### 環境の完全リセット
Docker環境を完全にクリーン化し、ゼロベースで再構築した。
```bash
docker rm -f $(docker ps -aq)
docker rmi -f $(docker images -q)
```
これにより「過去状態への依存」を完全に排除した。
### Dockerfileの再設計

以下をベースに再構築

- Ubuntu 24.04統一
- cron / jq / curl / git / tzdata 導入
- TZ=Asia/Tokyo 固定
- スクリプトを docker/ に集約
- entrypointによる起動時初期化
コンテナは永続環境ではなく
「破棄・再生成前提の実行単位」として設計した。
### UID問題の解消
従来
```bash
useradd -u 1000 r.h
```
→ ホストとのUID競合によりビルド失敗

修正後

```bash
useradd -m -s /bin/bash r.h
```

→ 環境依存を排除し再現性を優先

### entrypoint.sh の役割整理

entrypointは以下のみに限定

- crontab登録
- cronデーモン起動（cron -f）

アプリケーション本体ではなく
「初期化レイヤー」として分離した。
### docker-composeの整理

運用前提として以下を強化

- restart: unless-stopped
- env_fileで設定分離
- TZ=Asia/Tokyo固定
- working_dir固定
- volumesによる分離
- healthcheckでcron死活監視
- ログ分離（monitor / backup / cron）

→ 可観測性と再現性を強化

### 6. cron設計（状態管理型監視）

実行ジョブ

- 5分ごと：disk_monitor.sh
- 毎日9時：backup.sh

従来の問題
- 閾値ベースで通知が連発する
- 状態が分からない

改善後
- OK / WARN / CRITICAL の状態管理
- 状態変化時のみ通知
- 復旧通知あり
- state fileによる差分管理

→ 監視ノイズを構造的に削減

### 動作確認

- cronプロセス起動確認
- crontab反映確認
- コンテナ再生成後の再現性確認
- 同一構成での再ビルド成功

## 気づき

- cronは「設置」ではなく「起動設計」が必要
- Dockerは永続環境ではなく再生成前提で設計するべき
- entrypointは初期化専用レイヤーとして扱う
- UID固定は環境依存バグの原因になる
- 状態管理を入れることで監視設計の質が大きく変わる
- 責務分離が崩れると運用は必ず破綻する

## 成果

- ops-lab環境を完全再構築可能に整理
- cron + Dockerの設計分離を明確化
- 状態管理型監視の実装レベル理解
- 運用ノイズの削減設計を導入
- 単なる構築 → 運用設計へ移行
