# ops-lab CLI統合とアーキテクチャ再設計

ops-labプロジェクトの設計整理とCLI統合の完成、およびREADMEのアーキテクチャ構造をレイヤー設計として再構築。

## 実践内容

### ops_cli.shの統合完成

ops-labの運用を統合するCLIツール「ops_cli.sh」を完成させた。

このCLIは単なるショートカットではなく、運用レイヤーとして以下を統合している

- Docker操作の統合（up / down / restart / rebuild）
- コンテナ接続ショートカット（back / ops_shell）
- 監視系コマンド統合（status / health / logs）
- バックアップ・ディスク監視の実行
- Git操作統合（pull / repo確認）
- 障害調査・復旧支援（ops_diagnose / ops_fix）

これにより、手動で行っていた運用操作を1つのインターフェースに集約した。

### アーキテクチャのレイヤー再設計

従来の「処理の流れ（↓構造）」から、
「レイヤー構造」に設計を変更。

構造は以下の3層に整理

#### インフラ構築レイヤー
- Dockerfile
- docker-compose
- entrypoint.sh

→ 実行環境そのものを構築する層

#### 自動実行レイヤー
- cron
  - disk_monitor.sh
  - backup.sh

→ 定期的に自動で実行される運用処理層

#### 手動運用レイヤー
- ops_cli.sh

→ 人が直接操作する運用インターフェース層

この変更により、
「構築・自動・手動」が完全に分離された設計になった。

## 失敗・修正・解決したこと

### zshrc再読み込みによる関数競合・挙動不安定

`source ~/.zshrc` 実行時にCLIコマンドが一時的に不安定になる問題が発生

原因
- zshrc再読み込み時の関数定義順の問題
- backコマンドの定義と既存環境の影響

対応
- 一度シェルを再起動して環境をクリア
- ops_cli.shを再sourceして定義を統一

結果
- CLIコマンドが安定して動作するようになった

### CLIコマンドが認識されない問題

ops_helpやops_psが一時的に使えないことがあった

原因
- ops_cli.shのsourceがzshrcに反映されていない状態で実行していた

対応
- ~/.zshrcに以下を追加

000
source "$HOME/study/github/ops-lab/ops_cli.sh"
000

- シェル再読み込みで解決

結果
- 常時CLIコマンドが使える状態になった

### 環境設定とCLIの責務分離の理解

今回の作業で整理できたこと

- zshrc = 環境読み込み・設定管理
- ops_cli.sh = 運用機能本体
- aliasは極力使わず関数ベースで統一するのが安全

## 学び

- シェルエラーはコードではなく「読み込み順」で起きることが多い
- aliasとfunctionの混在はバグの原因になる
- CLIはzshrcに依存させすぎると壊れやすい
- 環境設定とツール本体は分離すべき設計思想
- トラブルシュートも設計理解の一部になる
