# restore.shの改善と安全な復元処理
## 学習内容
`restore.sh`の改善を行い、安全性・運用性を意識した復元処理を実装。
### 実装・改善内容
- バックアップ一覧表示機能の追加
- 最新バックアップ選択処理の改善
- ファイル名に含まれるタイムスタンプを利用した最新バックアップ判定
- 古すぎるバックアップの復元拒否機能
- 復元前スナップショット保存
- SHA256による整合性チェック
- ハッシュ不一致時の自動ロールバック
- LOCKファイルによる多重実行防止
- 復元前のdiff表示
- 実行確認プロンプト（FORCEモード対応）
- `restore_history.log`による復元履歴管理
- `restore_history.json`（JSON Lines形式）による履歴管理
- 復元回数のカウント管理
- Slack通知の改善
- エラーハンドリングの見直し
## 発生した問題
バックアップファイル名には日付が含まれているにもかかわらず、
```bash
ls -1t "$BACKUP_DIR"/index_*.html | head -1
```
で最新バックアップを取得すると、最も古いバックアップが選択されていた。

原因を調査したところ、`ls -t`はファイル名ではなく更新日時（mtime）で並び替えていることが原因。今回のバックアップファイルはコピーによって同じ更新日時になっていたため、ファイル名の日付順とは異なる結果になっていた。

## 改善内容
そのため、更新日時ではなくファイル名に含まれる日時で判定する方式へ変更。
```bash
LATEST=$(ls -1 "$BACKUP_DIR"/index_*.html | sort | tail -1)
```
さらに、古いバックアップを判定する処理も更新日時ではなく、ファイル名から日時を取得して経過時間を計算する方式へ変更。

## 学んだこと

今回の改善を通して、「最新ファイル＝更新日時が新しいファイル」とは限らないことを学びました。

復元処理では正常にコピーできるだけでは不十分であり、

- 復元前スナップショット
- SHA256による整合性確認
- ロールバック
- 多重実行防止
- 復元履歴管理
- 通知

など、安全に復元するための仕組みを組み合わせることが重要であることを学びました。
