# Linux学習記録 Day41-1
# ops-labのCLI化の進行
## 実践したこと

ops-labの操作を関数単位で直接呼び出す形から、opsコマンドを入口にしたCLI形式へ変更した。

今まではops_cli.shから関数を読み込み、直接コマンドを呼び出す形だったが、新しく「ops」というCLIの入口ファイルを作成した。

opsファイルではcase文を使い、入力されたコマンドによって処理を振り分ける仕組みを作った。

例
```bash
ops back
ops up
ops status
```

のように、コマンド名をシンプルにして実行できる形に変更した。

また、今まで使っていた関数は役割ごとのファイルに分けたまま維持した。

- shell.sh
  - コンテナ操作
- lifecycle.sh
  - 起動、停止、再構築
- status.sh
  - 状態確認、ログ確認
- script.sh
  - backup、execなど
- docker.sh
  - Docker操作
- git.sh
  - Git操作

というように、opsファイルは処理を書く場所ではなく、コマンドを振り分ける入口として使用した。

次にhelp.shを修正し、新しいCLI形式に合わせた。

以前
```bash
ops_up
ops_status
```

だった表示を、

```bash
up
status
```

のように変更し、実際に入力するコマンドと一致させた。

さらに、opsコマンドをどのディレクトリからでも使えるようにした。

```bash
sudo ln -s "$HOME/study/github/ops-lab/ops" /usr/local/bin/ops
```

でシンボリックリンクを作成し、

```bash
ops help
```

でホームディレクトリなど、ops-lab以外の場所からも実行できるようにした。

その際、最初はシンボリックリンク先の場所を基準にファイルを探してしまい、ops_cli.shが見つからない問題が発生した。

```bash
/usr/local/bin/ops: line 3: /usr/local/bin/ops_cli.sh: No such file or directory
```

というエラーが出た。

原因は、

```bash
source "$(dirname "$0")/ops_cli.sh"
```

がシンボリックリンク元ではなく、リンク先の場所を参照していたため。

修正後は実際のopsファイルの場所を取得するように変更した。

```bash
SCRIPT="$(readlink "$0")"

if [[ -z "$SCRIPT" ]]; then
    SCRIPT="$0"
fi

SCRIPT_DIR="$(cd "$(dirname "$SCRIPT")" && pwd)"

source "$SCRIPT_DIR/ops_cli.sh"
```

これにより、どこから実行しても正常に動作するCLIになった。

## 間違えたこと

- シンボリックリンクを作成しただけで、すぐにどこからでも動くと思っていた。

実際には、実行ファイル内で相対パスを使っていたため、リンク先の/usr/local/binを基準にファイルを探してしまった。

- opsファイルと各機能ファイルの役割を最初混同していた。

opsに全ての処理を書こうとしていたが、CLIの入口と実際の処理は分けるべきだと理解した。

opsは受付、cli内の各ファイルが処理担当という構造にした。

## 学んだこと

CLIツールは、単に処理を実行するだけではなく、使う人が分かりやすく操作できる設計が重要だと学んだ。

また、ファイルを役割ごとに分割することで、機能追加や修正がしやすくなることを理解した。

シンボリックリンクを使うことで、自作ツールをシステムコマンドのように扱えることを学んだ。

さらに、相対パスとシンボリックリンクの関係について理解できた。

見た目では同じファイルでも、実行方法によって基準となる場所が変わるため、パス設計は重要だと分かった。
