# Linux学習記録 Day43
# Docker環境改善とGitHub ActionsによるCI構築
## 実践内容

- Docker Composeのボリュームマウント設定を見直し、プロジェクト全体をコンテナと共有できる構成へ変更した。
- プロジェクト全体をマウントすることで、Mac側で編集したファイルがDockerコンテナ内にも即座に反映される開発環境を構築した。
- コンテナが正常に起動しなくなる問題が発生したため、`docker compose ps`と`docker compose logs`を使用して原因を調査した。
- `entrypoint.sh`の実行権限不足が原因でコンテナが再起動を繰り返していることを特定し、実行権限を修正して正常に起動できるようにした。
- cpu_monitor.shを作成し、CPU使用率の取得処理を実装・検証した。
- CPU使用率取得処理を実装する中で、macOSでは`top -l 1`、Linuxでは別のオプションを使用することを確認し、Dockerコンテナ（Linux）を基準に開発を進める方針へ変更した。
- GitHub Actionsを導入し、Push時に自動でCIが実行される環境を構築した。
- CIでは以下の品質チェックを自動実行するよう設定した。
  - Bash構文チェック
  - ShellCheck
  - Docker Build
- 初回はShellCheckの警告によってCIが失敗したため、ログを確認しながら原因を調査し、設定を修正して最終的にCIを正常に成功させた。

## 間違えたこと

- macOSとLinuxでtopコマンドの仕様が異なることを考慮せず実装を進めたため、Linuxコンテナ上でCPU監視スクリプトが正常に動作しなかった。
- Dockerコンテナへプロジェクト全体をマウントしたことで、`entrypoint.sh`の実行権限がホスト側の設定に依存することを理解しておらず、Permission deniedによってコンテナが起動しない問題を発生させた。
- GitHub ActionsのShellCheck設定を編集した際に`find`を二重に記述してしまい、不要なループを書いてしまった。
- ShellCheckの警告内容を十分に確認せず修正を進めたため、一度でCIを成功させることができなかった。

## 学んだこと

- Dockerではボリュームマウントによってホストとコンテナが同じファイルを共有できること。
- コンテナに問題が発生した場合は、`docker compose ps`と`docker compose logs`を利用して原因を調査することが重要であること。
- macOSとLinuxでは同じコマンドでも仕様が異なる場合があるため、本番環境を想定したLinux上で動作確認を行うことの重要性を学んだ。
- GitHub Actionsを利用することで、PushをトリガーにBash構文チェック・ShellCheck・Docker Buildを自動実行し、コード品質を継続的に確認できることを学んだ。
- ShellCheckはシェルスクリプトの問題点を早期に発見でき、保守性や品質向上に役立つことを理解した。
- CIはコードを書くだけではなく、品質を自動で確認する仕組みも含めて構築することが重要であることを学んだ。

## 振り返り

今日はDocker環境とCI環境の改善を中心に取り組んだ。特にGitHub Actionsを導入し、Pushするだけで自動的に品質チェックが実行される仕組みを構築できたことは大きな成果だった。また、macOSとLinuxのコマンド仕様の違いによる問題を経験し、開発環境だけではなく実際の実行環境を意識して検証することの重要性を学んだ。今後はCPU監視を完成させ、メモリ監視やHTTP監視なども追加し、より実運用を意識した監視ツールへ発展させていきたい。
