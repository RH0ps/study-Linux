# Linux学習記録 Day45
## CPU Monitor のOS差分対応・Linux環境検証
## 実践したこと

cpu_monitor.sh のOS依存部分を改善し、macOS・Linux両環境で動作可能な構成へ変更。

既存のCPU使用率取得処理では、macOS向けの `top -l 1` を使用していたため、Dockerコンテナ内のLinux環境で実行した際に `top: invalid option -- 'l'` エラーが発生。

原因調査として `uname` を実行し、DockerコンテナがLinux環境であることを確認。macOSの`top`コマンドとLinuxの`top`コマンドでは利用できるオプションや出力形式が異なるため、OSごとにCPU取得処理を分離する必要があると判断。

`get_cpu_usage()` 関数を追加し、`uname` の結果によって処理を分岐する構成へ変更。

macOSでは `top -l 1` からCPU idle値を取得し、Linuxでは `top -bn1` のCPU情報からidle値を抽出。取得したidle値を `awk` で計算し、CPU使用率へ変換する処理を実装。CPU取得処理を関数化することで、監視ロジックと環境依存処理を分離。

また、取得したCPU使用率に対して数値チェックと範囲チェックを追加。

000
if ! [[ "$USAGE" =~ ^[0-9]+$ ]] || (( USAGE < 0 || USAGE > 100 ))
000

不正な値や想定外のCPU使用率が取得された場合はログへ記録して終了するよう改善し、誤った監視結果を防止。

実装後、Mac側で `bash -n cpu_monitor.sh` を実行し、シェル構文チェックを実施。その後Dockerコンテナへ移動し、Linux環境で `bash cpu_monitor.sh` を実行して動作検証。

検証では以下の項目を確認。

- CPU使用率の取得
- 閾値による `OK` / `WARN` / `CRITICAL` 判定
- stateファイル更新
- Prometheus形式メトリクス出力
- 状態遷移ログ（OK → CRITICAL）の記録
- 状態変化時のSlack通知処理
- ログへの実行結果記録

実行結果として、

000
2026-07-13 20:15:03 [INFO] usage=100% state=CRITICAL
2026-07-13 20:15:03 [INFO] state: OK -> CRITICAL (slack=1)
000

が記録され、CPU使用率取得から状態判定、メトリクス出力、通知処理まで一連の監視フローが正常動作することを確認。

## 学んだこと

- OSごとに利用できるコマンドやオプション、出力形式が異なるため、環境依存部分を分離して実装することで可搬性を高められることを学んだ。
- DockerコンテナはLinux環境として動作するため、開発環境と実行環境の違いを考慮した検証が重要であることを学んだ。
- 監視スクリプトではCPU値を取得するだけでなく、不正値チェック、状態管理、メトリクス出力、通知処理まで含めて設計することで運用品質を向上できることを学んだ。
- 監視処理では正常動作だけでなく、異なる実行環境で発生する障害を想定し、原因調査、修正、再検証まで行うことが重要であることを学んだ。
