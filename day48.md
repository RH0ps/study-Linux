# Linux学習記録 Day48
# restore.shの完成とCPU監視スクリプトの改善
## 実践したこと
### cpu_monitor.sh
- `cpu_monitor.sh` の改善を行った。
- `PROJECT_ROOT` を利用したパス管理へ変更し、固定パスを減らして環境に依存しない構成へ修正した。
- `lib/log.sh` を利用するように変更し、ログ出力を共通化した。
- ログ・メトリクス・状態ファイルなどの保存先をプロジェクト構成に合わせて整理した。
- Slack Webhook が未設定でも動作するよう修正し、通知機能を有効・無効どちらでも実行できるよう改善した。
### restore.sh
- `restore.sh` の改善も行い、`PROJECT_ROOT` を利用したパス管理へ変更した。
- 復元前のスナップショット保存、ロールバック、SHA256ハッシュ検証、履歴保存、JSONログ出力などの処理を確認・調整した。
- `bash -n restore.sh` で構文チェックを実施した。
- `ShellCheck` を導入して静的解析を行い、スクリプトの品質を確認した。
- Dockerコンテナ内で `restore.sh` を実行し、バックアップ一覧取得、スナップショット作成、差分確認、ロールバック保存、復元、ハッシュ検証、履歴更新まで正常に動作することを確認した。

## 間違えたこと

- `BACKUP_DIR` のパスを誤って設定し、存在しないディレクトリを指定してしまった。
- `notify()` 関数の修正中に関数を二重定義してしまった。
- `log_warn` の記述でスペースが抜けていた。
- `ShellCheck` が未インストールの状態で実行しようとしてしまった。
- コマンドを `opa back` と入力し、正しくは `ops back` だった。
- `ks` と入力し、正しくは `ls` だった。

## 覚えたこと・コマンド

- `PROJECT_ROOT="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"`
  - スクリプト自身のディレクトリを取得できる。

- `bash -n restore.sh`
  - Bashスクリプトの構文チェックを行う。

- `shellcheck restore.sh`
  - Bashスクリプトの静的解析を行い、改善点を確認できる。

- `brew install shellcheck`
  - macOSへShellCheckをインストールできる。

- `diff ファイル1 ファイル2`
  - ファイル同士の差分を確認できる。

- `sha256sum`
  - ファイルのSHA256ハッシュ値を取得できる。

- `flock`
  - 多重実行を防止するためのロックを取得できる。

## 学んだこと

- `PROJECT_ROOT` を利用してパスを管理すると、実行場所に依存しないスクリプトを作成できる。
- ログ処理を共通ライブラリへ分離すると、複数のスクリプトで同じ処理を再利用でき、保守性が向上する。
- CPU監視や復元スクリプトなど複数のスクリプトで設計を統一すると、修正や機能追加がしやすくなる。
- Slack通知は必須にせず、環境変数が設定されている場合のみ有効化する設計にすると柔軟性が高くなる。
- `ShellCheck` を利用することで、構文エラーだけでなく保守性や可読性の問題も確認できる。
- 実際にDockerコンテナ内で動作確認を行うことで、想定どおりにスクリプトが動作することを確認できた。
